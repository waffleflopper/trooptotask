# Organization Hierarchy (Parent-Child Orgs) — Design Spec

**Date**: 2026-03-13
**Status**: Draft

---

## Overview

Enable military organizational hierarchy by allowing a parent org (e.g., Battalion) to link to child orgs (e.g., Companies). The parent gets read-only visibility into child org data and can push standardized type definitions down to children. Each org remains independently operated and billed.

### Key Decisions

- **Single-level hierarchy** — parent → children only, no grandchildren
- **Invitation model** — child shares a code, parent uses it to invite, child accepts
- **Independent billing** — no billing entanglement between parent and child
- **Read + limited control** — parent views child data (including drill-down to individuals) and pushes type definitions
- **No tier gating** — available to all subscription tiers

## Data Model

### New Table: `org_relationships`

| Column | Type | Notes |
|--------|------|-------|
| `id` | uuid | PK, default `gen_random_uuid()` |
| `parent_org_id` | uuid FK → organizations | The parent org |
| `child_org_id` | uuid FK → organizations | The child org |
| `status` | text check (`pending`, `active`, `declined`, `severed`) | Relationship state |
| `invited_at` | timestamptz | When parent sent the invite |
| `invited_by` | uuid FK → auth.users | Who in parent org sent it |
| `accepted_at` | timestamptz | Nullable, set on acceptance |
| `accepted_by` | uuid FK → auth.users | Who in child org accepted |
| `severed_at` | timestamptz | Nullable, set on severance |
| `severed_by` | uuid FK → auth.users | Who severed the link |

**Constraints:**
- Unique on `child_org_id` WHERE `status = 'active'` (one active parent only)
- Check: `parent_org_id != child_org_id`
- **Single-level enforcement:** An org with an active relationship as a child cannot be a parent, and an org with active relationships as a parent cannot be a child. Enforced at application level on invite creation (check both directions before allowing).
- Severed rows are kept for audit history; re-linking creates a new row

### Modified Table: `organizations`

Add `link_code` column:
- Type: text, unique, not null, default `encode(gen_random_bytes(6), 'hex')` (12-char hex string)
- Generated at org creation
- Displayed in Admin area for sharing
- Can be regenerated by admin/owner (invalidates old code). On the rare unique constraint collision during generation, retry with a new random value.
- Used by parent org to identify the child when sending an invite

### New Table: `shared_type_configs`

Tracks which of the parent's types are currently shared with children.

| Column | Type | Notes |
|--------|------|-------|
| `id` | uuid | PK, default `gen_random_uuid()` |
| `parent_org_id` | uuid FK → organizations | The parent org |
| `type_category` | text check (`training`, `status`, `assignment`, `counseling`) | Which type table |
| `type_id` | uuid | The parent's type ID being shared |
| `created_at` | timestamptz | When sharing was enabled |

**Constraints:**
- Unique on `(parent_org_id, type_category, type_id)`
- Deleting a row from this table triggers unsharing (orphans copies in children)

### Modified Tables: Type Tables

Add to `training_types`, `status_types`, `assignment_types`, `counseling_types`:
- `source_org_id` (nullable uuid FK → organizations) — which parent org pushed this type
- `source_type_id` (nullable uuid) — the original type ID in the parent org (used for propagating edits)

Both columns are null for locally created types. Both are set when a type is pushed from a parent. Index on `(source_org_id, source_type_id)` for efficient propagation queries.

Rules:
- `NULL` source = locally created
- Non-null source = pushed from that parent org
- Child org cannot edit or delete rows where `source_org_id IS NOT NULL`
- On severance or unsharing, set both `source_org_id` and `source_type_id` to `NULL` (orphan in place)

### No Changes To

- `organization_memberships` table
- `personnel`, `groups`, or any other existing tables

## Invitation & Relationship Workflow

### Invite Code

Each org has a shareable link code displayed in their Admin area (e.g., "Your org link code: `ABC123`"). The child org admin shares this out-of-band (email, Slack, etc.) with the parent org admin.

### Flow

1. **Parent invites child** — Parent admin/owner enters the child's invite code on the Subordinate Units page. New `org_relationships` row created with `status: pending`.
2. **Child notified** — Notification sent to child org admins/owners via the existing notification system.
3. **Child accepts** — Child admin/owner accepts from their Admin area. Status becomes `active`, `accepted_at`/`accepted_by` set. All currently-shared parent types are pushed to the child.
4. **Child declines** — Status set to `declined`. Row kept for audit history.
5. **Severance** — Either side's admin/owner can sever. Status becomes `severed`, `severed_at`/`severed_by` set. All pushed types in child get `source_org_id` and `source_type_id` set to NULL. Notification sent to the other org's admins. Note: `shared_type_configs` rows for the parent are preserved — they represent the parent's sharing intent, not per-child state. If the parent links a new child later, those types are still marked as shared.
6. **Re-linking** — Follow the same flow from step 1. Creates a new row; the old severed row remains for history.

### Permission Gating

| Action | Who |
|--------|-----|
| Send invite | Parent admin/owner |
| Accept/decline invite | Child admin/owner |
| Sever from parent side | Parent admin/owner |
| Sever from child side | Child admin/owner |

## Pushed-Down Types

### How Pushing Works

Parent admin/owner manages "Shared Types" on the Subordinate Units page. They select which of their own training types, status types, assignment types, and counseling types to share with all children. Sharing a type creates a row in `shared_type_configs`.

- When a type is marked as shared → a copy is created in every active child org's type table with `source_org_id` and `source_type_id` set. The copy is an independent row in the child's type table, owned by the child org's `organization_id`.
- When a new child accepts the invite → all types listed in `shared_type_configs` for the parent are copied to the child.
- When a new type is shared → copies are created in all active children immediately.

### Edit Propagation

When a parent edits a shared type (name, color, etc.), the update propagates to all child copies synchronously in the same API call using service role. The query: `UPDATE [type_table] SET name = ..., color = ... WHERE source_type_id = [parent_type_id] AND source_org_id = [parent_org_id]`. This is a single query regardless of child count. If the update fails for any reason, the parent edit is rolled back (transaction).

### Editing & Deletion Rules

| Action | Parent | Child |
|--------|--------|-------|
| Edit a pushed type (name, color, etc.) | Yes — change propagates to all children | No — read-only |
| Unshare a type | Yes — orphans in children (`source_org_id` → NULL) | N/A |
| Delete pushed type while shared | Not allowed — must unshare first | No — can't delete while linked |
| Add local types alongside pushed ones | N/A | Yes — full control of their own types |

### Visual Distinction

In child org type managers, pushed types display a Badge or indicator showing the parent org name, making it clear the type is externally managed and cannot be edited locally.

### Type Merge Tool

A general-purpose tool available to any org's admin/owner for consolidating one type into another within the same type category:

1. Admin selects a source type and chooses "Merge into..." → picks a target type (must be same category — e.g., training type into training type)
2. Confirmation dialog shows how many records will be reassigned
3. All records referencing the source type are updated to reference the target type (within a transaction)
4. The source type is deleted

**Validation:** Source and target must be the same type category (training, status, assignment, counseling). The API endpoint enforces this. Merging across categories is not allowed.

This handles naming collisions (parent pushes "Weapons Qualification", child already has "Weapons Qual") and any other case where types should be consolidated. The merge tool is not limited to same-name types, and is useful even outside parent-child relationships for general type cleanup.

## Parent Visibility — Subordinate Units Page

### Navigation

New nav item "Subordinate Units" visible to admin/owner. The page handles the empty state gracefully (no children yet → shows invite code + instructions for linking). Shows:
- The invite code management and invite workflow
- List of child orgs (if parent)
- Pending invites (if any)

### Top Level: Child Org Cards

Each active child org displayed as a summary card:
- Org name
- Personnel headcount
- Training compliance summary — calculated across **all** training types in the child org (both pushed and local). Percentage = personnel with current records / total personnel, averaged across all types.
- Today's availability summary (e.g., "112 present, 5 leave, 3 TDY")
- Link to drill-down view

### Drill-Down: Child Org Detail View (Read-Only)

Organized into tabs or sections:
- **Personnel** — Full roster (name, rank, group) with status indicators
- **Training** — Compliance matrix view (who's current, who's overdue, on which types)
- **Calendar/Availability** — Current status snapshot and upcoming statuses

All content is strictly read-only. No edit capabilities from the parent side.

### Data Fetching

Server-side queries against child org data, authorized by confirming:
1. An active `org_relationships` row exists between parent and child
2. The requesting user is admin/owner in the parent org

Existing `orgId`-scoped query patterns are reused, called with the child's `orgId`.

### Audit Logging

**All drill-down access is audit-logged in both the parent and child org logs:**

| Field | Value |
|-------|-------|
| `org_id` | Logged in **both** parent and child org |
| `action` | `parent_accessed_child_data` |
| `actor` | The parent org user who drilled in |
| `details` | Which child org, what section accessed (personnel/training/calendar) |

This ensures child org admins can see who from higher has accessed their data, and parent org has a record of cross-org access by their users.

- Pushed types appear in type managers with parent org badge (read-only)
- Merge tool available for consolidating local types into pushed types
- Pending parent invites appear via notification system and Admin area
- No other UI changes — the app works exactly the same

## Edge Cases

### Parent org deleted or archived
Sever all active child relationships. Orphan all pushed types in children (`source_org_id` → NULL). Children continue operating independently.

### Child org deleted or archived
Relationship status becomes `severed`. Parent's Subordinate Units page no longer shows them.

### Name collision on push
Parent pushes a type with the same name as an existing child type. Both coexist — one marked as local, one marked as pushed. Child admin can use the merge tool to consolidate.

### User is admin/owner in both parent and child orgs
Both perspectives work normally. No special handling needed.

### Re-linking after severance
Same invite flow as initial linking. New row in `org_relationships`. Previously orphaned types remain local in the child; the parent's currently-shared types are pushed fresh.

### Concurrent operations during severance
Severance (status update + type orphaning) is performed in a single database transaction. Any in-flight parent type edits that hit after the transaction commits will find no matching `source_type_id`/`source_org_id` rows and become no-ops.

## RLS Considerations

- `org_relationships`: Parent org members can read rows where `parent_org_id` matches their org. Child org members can read rows where `child_org_id` matches their org. Only admin/owner can insert/update.
- Cross-org data access (parent viewing child data): Server-side queries using service role, gated by application-level permission checks (active relationship + admin/owner in parent). Do NOT rely solely on RLS for cross-org access.
- Pushed types: Standard type table RLS applies. `source_org_id` is informational; edit/delete restrictions enforced at application level.

## API Endpoints

### Relationship Management
- `POST /org/[orgId]/api/subordinate-units/invite` — Send invite (parent admin/owner)
- `POST /org/[orgId]/api/subordinate-units/accept` — Accept invite (child admin/owner)
- `POST /org/[orgId]/api/subordinate-units/decline` — Decline invite (child admin/owner)
- `POST /org/[orgId]/api/subordinate-units/sever` — Sever relationship (either side admin/owner)

### Shared Types
- `GET /org/[orgId]/api/subordinate-units/shared-types` — List parent's shared types
- `POST /org/[orgId]/api/subordinate-units/shared-types` — Share/unshare types (parent admin/owner)

### Type Merge
- `POST /org/[orgId]/api/types/merge` — Merge one type into another within the same category (admin/owner, any org)

### Child Data Access (Parent)
- `GET /org/[orgId]/api/subordinate-units/[childOrgId]/summary` — Child org summary card data
- `GET /org/[orgId]/api/subordinate-units/[childOrgId]/personnel` — Child personnel roster
- `GET /org/[orgId]/api/subordinate-units/[childOrgId]/training` — Child training compliance
- `GET /org/[orgId]/api/subordinate-units/[childOrgId]/availability` — Child availability/status

All child data endpoints log to audit in both parent and child orgs.

---

## Security

- All API endpoints include server-side permission checks (admin/owner + active relationship)
- All user input validated/sanitized via `src/lib/server/validation.ts`
- All cross-org data access audit-logged
- Rate limiting applied to all new endpoints; cross-org data access endpoints (child summary, personnel, training, availability) should have stricter rate limits than standard endpoints due to heavier query load
- Invite codes: 12-character hex strings generated via `encode(gen_random_bytes(6), 'hex')` — 48 bits of entropy, regenerable by admin/owner
- No PII logged in audit details (log org IDs and section names, not personnel names)
